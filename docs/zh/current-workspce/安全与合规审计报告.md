# 代码库安全与合规审计报告

## 执行摘要

本次审计在 campus-energy-optimizer 代码库中识别出若干关键安全漏洞和代码质量问题。最严重的问题是 **`.env` 文件中的硬编码 API 密钥**，如果提交到版本控制系统将构成严重的安全风险。其他问题包括潜在的 N+1 查询模式、低效的 React 渲染以及缺少身份验证/授权检查。

## 关键安全问题

### 1. 版本控制中的硬编码 API 密钥
**文件**: `/home/zch/test/.env`
**严重程度**: 严重

`.env` 文件包含绝不应该提交到版本控制的实际 API 密钥：
- `LLM_API_KEY='sk-79f35e625cbd43c8b4f22836eaefdd43'`
- `LANGFUSE_SECRET_KEY="sk-lf-fe67f559-ac43-4e8b-92ef-9ef9f988966b"`
- `LANGFUSE_PUBLIC_KEY="pk-lf-403af868-6437-4377-a0f7-5ddd337b29e0"`

**建议措施**： 
- 立即轮换这些 API 密钥
- 将 `.env` 添加到 `.gitignore`
- 在部署流水线中使用环境变量管理
- 在 CI/CD 中实施密钥扫描

### 2. 缺少身份验证/授权
**文件**: `/home/zch/test/src/server/actions/` 中的所有服务器操作
**严重程度**: 高

`generate-plan.ts`、`save-course-details.ts` 等服务器操作缺少任何身份验证或授权检查。这允许未经身份验证的用户可能：
- 生成计划
- 修改课程详情
- 访问用户数据
- 执行任意操作

**建议措施**：
- 为所有服务器操作实现身份验证中间件
- 添加基于角色的访问控制
- 在数据修改前验证用户权限

## 性能问题

### 1. N+1 查询模式
**文件**： 
- `/home/zch/test/src/lib/repo/course-details.ts`
- `/home/zch/test/src/lib/repo/timetable.ts`

两个文件都在循环内使用顺序数据库操作而不是批量操作：

```typescript
// 当前存在问题的模式
for (const detail of details) {
  await tx.courseDetail.create({ data: { ... } })
}

// 推荐修复方案
await tx.courseDetail.createMany({ 
  data: details.map(detail => ({ ... }))
})
```

**影响**：由于多次数据库往返，大数据集会出现显著的性能下降。

### 2. 低效的 React 渲染
**文件**: `/home/zch/test/src/components/plan/plan-table.tsx`
**问题**: 助手函数 `buildShortTermSummary` 和 `buildDetailLines` 在组件内部定义，导致每次渲染时都会产生不必要的重渲染和函数分配。

**建议措施**: 将助手函数移出组件外部，或使用 `useMemo`/`useCallback` 包装。

## 错误处理问题

### 1. 通用错误处理
多个文件使用通用的 `throw new Error()` 模式，没有结构化错误类型，使得在 UI 中难以适当地处理错误。

**文件**: 各种服务器操作和工具函数
**建议措施**: 创建具有特定错误代码的自定义错误类，以实现更好的错误处理和用户反馈。

### 2. 缺少错误边界
React 组件缺少适当的错误边界，可能导致整个应用程序在渲染错误时崩溃。

**建议措施**: 在适当级别实施错误边界（路由级和组件级）。
